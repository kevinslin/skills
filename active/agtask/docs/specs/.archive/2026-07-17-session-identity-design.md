# Feature Design: Creation Identity and Codex Session Binding

**Date:** 2026-07-17
**Status:** Implemented locally; not published
**Owner:** agtask

## Goal

Give each agtask creation attempt a logical, non-secret UUID before Codex creates
the child, then bind that logical **thread.id** to exactly one real Codex
**thread.session_id**. A bootstrap copied into Codex's internal title-generation
session must not create a second task or emit title/pin actions.

## Scope

In scope:

- Make **thread.id** a pre-creation logical UUID and add unique
  **thread.session_id** for the real Codex session UUID.
- Rename **parent_thread_id** to **parent_session_id** on every agtask-owned
  storage, command, JSON, and UI contract.
- Add logical **id** to the strict canonical version-2 child bootstrap.
- Bind **id -> session_id** transactionally across hook-first, parent-first, and
  bootstrap-fallback orderings.
- Reject copied bootstraps after the ID is bound to another session, with no row,
  rollout, tracked context, title action, or pin action.
- Update CLI, hooks, lifecycle paths, dashboard/detail/deep links, tests,
  integration scenarios, and synchronized documentation.

Out of scope:

- Treating either UUID as a credential. The creation ID is a correlation nonce,
  not a secret or authorization token.
- Giving **parent_session_id** a foreign key; the parent may be untracked.
- Changing Codex-owned names such as hook **session_id**, app API **threadId**,
  or delegation wrapper **source_thread_id**.
- Automatically migrating an older ledger. This is a hard-cut schema version.

## Observed Failure

One child creation produced two rows:

| Role | Session ID | Stored description |
| --- | --- | --- |
| Real child | **019f6e67-dbc6-7111-b791-43d223817140** | “Implement a simple per-project merge queue for agtask.” |
| Codex title generator | **019f6e67-e372-7d03-bd30-4afcbe485556** | “You are a helpful assistant.” |

Both had the same parent and title. The second session returned:

~~~json
{"title":"Implement per-project merge queue","description":"Add SQLite-backed agtask merge serialization integrated with OnPreClose lifecycle"}
~~~

About 150 milliseconds later, that title was assigned to the real child. This
identifies the second session as Codex's title generator. The global
UserPromptSubmit hook accepted the copied final v2 bootstrap, assigned its
internal thread ID directly from the hook payload session ID, and registered
any untracked session. Titles are not unique, so SQLite accepted both rows.

## Implemented State

Schema version 5 implements the selected identity split:

- **thread.id** is the canonical UUIDv4 creation ID generated by
  **resolve-create** before Codex creates or forks a child.
- **thread.session_id** is the unique real Codex session ID.
- **thread.parent_session_id** records the invoking Codex session and remains a
  non-foreign-key lineage field.
- **rollout.thread_id** and
  **project_merge_claim.owner_thread_id** continue to reference logical
  **thread.id**.
- The strict v2 bootstrap requires **id** and **parent_session_id**.
- The UserPromptSubmit hook binds **id -> session_id** under
  **BEGIN IMMEDIATE** and rejects copied or conflicting identity pairs without
  creating a row, rollout, tracked context, title action, or pin action.
- Explicit task commands resolve mutually exclusive **--id** and
  **--session-id** selectors to the same logical row. Codex actions, task
  routes, and deep links use **session_id**.

## Requirements -> Design Mapping

| Requirement | Design decision |
| --- | --- |
| Identify one creation across copied prompts | resolve-create generates one canonical UUIDv4 **id** and puts it in the v2 bootstrap. |
| Store real Codex identity | Add nonempty unique **thread.session_id TEXT**. |
| Preserve rollout ownership | Keep **rollout.thread_id** as the foreign key to logical **thread.id**. |
| Name parent identity correctly | Rename agtask-owned **parent_thread_id** to **parent_session_id**; keep it non-FK. |
| Reject title-generator copies | Under **BEGIN IMMEDIATE**, an ID bound to another session is a silent copied-bootstrap no-op. |
| Preserve retries | The same **(id, session_id)** pair reconciles idempotently. |
| Converge parent and hook registration | Parent register supplies both **--id** and **--session-id** and uses the same pair rules. |
| Keep Codex navigation correct | Hooks, app actions, routes, and deep links use **session_id**, never logical **id**. |
| Make compatibility explicit | Hard-cut schema version 4 to schema version 5. |
| Make results unambiguous | Thread JSON exposes both IDs; rollout **thread_id** remains logical ID. |

## Implemented Design

### 1) Identity and schema

The invariant is: **id** identifies one agtask creation attempt;
**session_id** identifies the one Codex session bound to it.

resolve-create uses Python uuid.uuid4(). The accepted form is lowercase,
hyphenated, canonical UUIDv4. It is random to avoid creator coordination, but
it grants no authority.

~~~sql
CREATE TABLE thread (
  id TEXT PRIMARY KEY NOT NULL,
  session_id TEXT NOT NULL,
  parent_session_id TEXT,
  kind TEXT NOT NULL,
  project TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT NOT NULL DEFAULT '',
  created TEXT NOT NULL,
  updated TEXT NOT NULL,
  closed TEXT,
  status TEXT NOT NULL,
  CHECK (id <> ''),
  CHECK (session_id <> ''),
  CHECK (parent_session_id IS NULL OR parent_session_id <> ''),
  CHECK (parent_session_id IS NULL OR parent_session_id <> session_id),
  CHECK (length(trim(project)) > 0),
  CHECK (kind IN ('main', 'child')),
  CHECK (
    (kind = 'main' AND parent_session_id IS NULL) OR
    (kind = 'child' AND parent_session_id IS NOT NULL)
  ),
  CHECK (status IN ('todo', 'active', 'blocked', 'merging', 'done')),
  CHECK (
    (status = 'done' AND closed IS NOT NULL) OR
    (status <> 'done' AND closed IS NULL)
  )
);

CREATE UNIQUE INDEX thread_session_id_idx ON thread(session_id);
CREATE INDEX thread_parent_session_idx ON thread(parent_session_id);
CREATE INDEX thread_created_idx ON thread(created);
CREATE INDEX thread_status_updated_idx ON thread(status, updated);
CREATE UNIQUE INDEX thread_project_merging_idx
  ON thread(project) WHERE status = 'merging';

CREATE TABLE project_merge_claim (
  project TEXT PRIMARY KEY NOT NULL CHECK (length(trim(project)) > 0),
  owner_thread_id TEXT NOT NULL REFERENCES thread(id) ON DELETE CASCADE,
  token TEXT NOT NULL,
  acquired TEXT NOT NULL,
  heartbeat TEXT NOT NULL,
  lease_expires TEXT NOT NULL,
  prior_status TEXT NOT NULL CHECK (prior_status IN ('todo', 'active', 'blocked'))
);

CREATE UNIQUE INDEX project_merge_claim_owner_idx
  ON project_merge_claim(owner_thread_id);
CREATE UNIQUE INDEX project_merge_claim_token_idx
  ON project_merge_claim(token);

CREATE TABLE rollout (
  id INTEGER PRIMARY KEY,
  created TEXT NOT NULL,
  thread_id TEXT NOT NULL REFERENCES thread(id) ON DELETE CASCADE,
  turn_id TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'meta')),
  message TEXT NOT NULL
);
~~~

The implementation preserves the version-4 rollout order/role-aware unique
indexes, FTS table and triggers. FTS continues to join through SQLite rowid;
neither UUID becomes the FTS content rowid. Merge claims continue to refer to
the logical **thread.id**, so all merge-slot and tokenized-close operations
resolve an explicit selector to logical ID before reading or changing a claim.

Thread JSON:

~~~json
{
  "id": "1d7e4ef1-0c28-4a3d-93b1-779c7fe52bd8",
  "session_id": "019f6e67-dbc6-7111-b791-43d223817140",
  "parent_session_id": "019f6e68-19d6-7e10-a449-506a0de88117"
}
~~~

Rollout JSON retains the logical owner:

~~~json
{"id":108,"thread_id":"1d7e4ef1-0c28-4a3d-93b1-779c7fe52bd8","turn_id":"019f6e67-...","role":"user","message":"Implement a simple per-project merge queue for agtask."}
~~~

### 2) Canonical creation protocol

resolve-create generates an ID for every task designation. For a child it
returns the ID and includes it in the strict final v2 trailer:

~~~text
<agtask-bootstrap version="2">
{"id":"1d7e4ef1-0c28-4a3d-93b1-779c7fe52bd8","parent_session_id":"019f6e68-19d6-7e10-a449-506a0de88117","pin":true,"project":"agtask","title":"agtask/merge-queue"}
</agtask-bootstrap>
~~~

The JSON keeps exact allowlisted keys, deterministic key order and encoding, no
duplicates or extra whitespace, and final-trailer position. ID is required and
must be canonical UUIDv4. **parent_session_id** replaces
**parent_thread_id** in the agtask-owned v2 contract.

The parent reuses the exact resolver ID for the entire creation attempt. After
Codex returns the real child threadId, it registers:

~~~sh
agtask register --id 1d7e4ef1-0c28-4a3d-93b1-779c7fe52bd8 --session-id 019f6e67-dbc6-7111-b791-43d223817140 --parent-session-id 019f6e68-19d6-7e10-a449-506a0de88117 ...
~~~

Main designation also gets a resolver-generated logical ID and registers it
with the invoking session. Its v1 action trailer need not carry ID because main
registration does not use child bootstrap fallback.

Each new resolve-create call is a new attempt and generates a new ID. Retrying
downstream work for one attempt must reuse the returned ID rather than call the
resolver again.

### 3) Hook classification and transactional binding

UserPromptSubmit first resolves the payload **session_id** to determine whether
the hook belongs to an already tracked task, then independently validates the
complete bootstrap before rendering any bootstrap app-action context. An absent
bootstrap is classified with invalid bootstrap for this decision:

| Payload session | Bootstrap | Required behavior |
| --- | --- | --- |
| Tracked | Invalid or absent | Ignore the bootstrap, record the ordinary rollout by the row's logical ID, return ordinary tracked context, and commit. Emit no bootstrap registration, title action, or pin action. |
| Tracked | Valid | Enter the exact tracked-pair matrix below. Do not record the ordinary rollout or render any context until identity and immutable metadata both reconcile. |
| Untracked | Valid | Enter the transactional binding state machine below. Only a successful unclaimed or exact same-pair result may return tracked context or bootstrap actions. |
| Untracked | Invalid or absent | Fail open silently: no registration, rollout, tracked context, title action, or pin action. |

For a tracked payload session with a valid bootstrap, the transaction reads
both the row selected by **session_id** and the row selected by bootstrap
**id**. Immutable registration metadata means the inferred child **kind**,
**parent_session_id**, **project**, **title**, and the initial **description**
derived from the normalized prompt; the first four are bootstrap-addressable.
The cases are intentionally exhaustive:

| Stored identity and bootstrap | Writes | Returned context / actions | Transaction result |
| --- | --- | --- | --- |
| Bootstrap **id** and payload **session_id** name the same row, and all immutable registration metadata matches | Reconcile the turn through the ordinary idempotent rollout path; write only a genuinely new rollout. | Return ordinary tracked context; render the title action and render the pin action exactly when **pin=true**. | Commit. |
| The same exact pair is named, but immutable metadata or title conflicts | None, including no ordinary rollout and no metadata update. | No output, tracked context, title action, or pin action. | Roll back/read-only no-op. This preserves the current registration-conflict contract. |
| The payload session is tracked to row A, while bootstrap **id** is already owned by row B | None, including no ordinary rollout. | No output, tracked context, title action, or pin action. | Roll back/read-only no-op; classify as copied or conflicting bootstrap. |
| The payload session is tracked to row A, while bootstrap **id** is unclaimed | None; never create or rebind a row because the session is already owned by another logical ID. | No output, tracked context, title action, or pin action. | Roll back/read-only no-op; classify as session-owned-by-another-ID conflict. |

Consequently, a valid bootstrap is not harmless decoration on an ordinary
tracked turn: if it conflicts, the whole hook attempt is silent and atomic.
Only invalid or absent bootstrap text is ignored while the ordinary tracked
turn proceeds.

For an untracked session with valid v2 input, binding runs under
**BEGIN IMMEDIATE**:

1. Parse **id**, **parent_session_id**, **project**, **title**, and **pin**.
2. Read candidate Codex identity from **payload.session_id**.
3. Query thread by logical ID.
4. If ID is unclaimed, verify session ID is also unclaimed, insert the pair,
   append thread.created and the real user rollout under logical ID, commit, and
   then render title/pin actions for session ID.
5. If ID is bound to the same session, verify all immutable registration
   fields, including title and normalized initial description. On an exact
   metadata match, reconcile the turn through existing idempotency logic,
   commit, return tracked context, render the title action, and render the pin
   action exactly when **pin=true**. On metadata/title/description conflict,
   write nothing, return no output or actions, and roll back/read-only no-op.
6. If ID is bound to another session, classify it as copied/internal bootstrap;
   roll back or perform a read-only no-op and emit no row, rollout, tracked
   context, title action, or pin action.
7. If session ID belongs to another logical ID, treat it as identity conflict,
   roll back, and emit no actions.

Primary-key uniqueness on ID and unique session ID are race backstops.
Constraint errors are re-read and classified through the same state machine;
they cannot leak partial writes or generic action context.

Parent registration uses the same semantics:

| Stored state | Requested pair | Result |
| --- | --- | --- |
| Neither value owned | New pair | Insert |
| ID owns same session | Same pair | Idempotent reconcile |
| ID owns another session | Different session | Conflict; preserve row |
| Session belongs to another ID | Different ID | Conflict; preserve row |

Thus parent-first, hook-first, and parent-after-bootstrap paths converge on one
row. Existing immutable kind, project, parent, and description conflicts still
apply after identity convergence, as does immutable title conflict.

### 4) Ordinary hooks and lifecycle

After creation, UserPromptSubmit, Stop, PostCompact, and SessionStart query
**thread WHERE session_id=?**, retrieve logical ID, and pass logical ID to
rollout and transition helpers.

Naming is normative:

- **id** or **thread_id** means logical creation ID.
- **session_id** means real Codex session ID.
- **parent_session_id** means invoking Codex session ID.
- No variable named thread ID may contain payload session ID.

Codex-owned hook session_id, app action threadId, creation response threadId,
and codex://threads/<session-id> retain platform vocabulary. The adapter maps
them to **thread.session_id**.

### 5) CLI, JSON, and dashboard

register requires both identity inputs, **--id** and **--session-id**;
resolve-create returns ID, and child creation accepts **--parent-session-id**.
Registration is pair binding, not selector-based lookup.

Every explicit thread-targeting command, including show, status, reopen, close
and its merge-claim actions, append-rollout, and record-turn, uses one shared
selector resolver with a mutually exclusive required **--id** / **--session-id**
argument group. The resolver returns the logical row and canonical logical ID;
commands persist and own claims/rollouts by logical ID. Hooks never use this
CLI selector abstraction: they always enter through payload **session_id** and
resolve it to the logical row.

List and search need no selector. Tracked thread results expose **id**,
**session_id**, and **parent_session_id**. Rollouts keep logical
**thread_id**. An untracked
**close --if-tracked --session-id ... --json** result returns session ID and
untracked status, but cannot fabricate logical ID.

Old **--id <codex-session-id>** use is removed, not aliased. ID always means the
logical creation ID.

Dashboard list/detail JSON exposes both IDs. Stable task data and rollout
ownership use ID. Routes, Codex links, title/pin actions, and session labels use
session ID. Parent facets use **parent_session_id**.

Human-readable output uses these exact labels:

- **Task ID:** logical **thread.id**.
- **Session ID:** Codex **thread.session_id**.
- **Parent session ID:** Codex **thread.parent_session_id**.

All displayed/opened links use Session ID.

~~~text
/tasks/~<encoded-session-id>
/api/tasks/~<encoded-session-id>
codex://threads/<encoded-session-id>
~~~

Server and client code name the decoded route value session ID and query that
column. No logical ID may enter a Codex URL or app action.

## Ordering Dependency and Failure Policy

The implementation depends on the real child's prompt hook claiming an
unbound creation ID before any internal Codex session can submit a copied
bootstrap. The supplied production trace is the accepted external platform
evidence and contract for that dependency: the real child session
**019f6e67-dbc6-7111-b791-43d223817140** ran first, the internal title-generator
session **019f6e67-e372-7d03-bd30-4afcbe485556** ran afterward, and the generated
title was assigned to the real child about 150 milliseconds later.

The repository's standalone integration harness cannot create the desktop
internal title-generator session and cannot prove host scheduling order for
one-shot or queued creation. It must not label a synthetic second hook, a
clientThreadId transition, or timing observed inside the harness as live proof
of platform ordering. Repository verification instead requires an exact,
deterministic replay of the supplied two-session trace through the binding
state machine, plus all same-pair, metadata-conflict, ID-conflict, and
session-conflict cases. That proves behavior given the platform contract; it
does not prove the external contract itself.

Deployment and release remain contingent on primary-before-copy platform
behavior. The supplied trace satisfies the current release input; any contrary
host evidence from a supported surface blocks release of the entire unclaimed-ID
fallback. It must then be redesigned around either:

1. Two-phase creation: create/fork an empty child, receive session ID, register
   the pair, then send the task prompt.
2. A Codex-provided uncopyable signal identifying the primary task session.

This policy also covers queued creation that initially returns only
**clientThreadId**. Never guess the real session from title, description,
timing, or UUID order. There is no runtime or per-surface switch for enabling
the unclaimed-ID fallback.

## Compatibility and Schema Version

This change rebased onto schema version 4 from commit **0cdacb4** and shipped
locally as hard-cut schema version **5**.

- Empty ledger initializes directly at the selected version.
- Older/mismatched ledger is rejected without mutation, preserving bytes/mode.
- No automatic migration, dual read/write, compatibility view, or fallback
  interpretation of old thread ID exists.
- Old v2 trailers missing ID, using parent_thread_id, using malformed/non-v4 ID,
  or failing canonical encoding are invalid bootstraps. An untracked session
  fails open with no registration, rollout, or actions; a tracked session still
  follows its ordinary rollout/context path but emits no bootstrap actions.
- Parent emitter and child hook parser update together.
- Version-4 merge serialization is retained: **merging**, the partial unique
  project/status index, merge claim table and indexes/FKs, logical
  **owner_thread_id**, lease/token behavior, and tokenized close all remain.

Any future old-ledger preservation requires a separately specified offline
migration.

## Parity Audit

Preserve:

- Fail-open behavior for absent ledger, untracked ordinary session, unsafe
  payload, incompatible schema, and locked database.
- Atomic registration, thread.created, and initial user rollout.
- Rollout replay/conflict rules and role-aware uniqueness.
- Immutable kind, project, parent lineage, and initial description.
- Main-null/child-required parent semantics with untracked parents allowed.
- Status (including **merging**), done, merge-claim leases/tokens, tokenized
  close prompts, timestamps, compaction, FTS order, and read-only dashboard
  behavior.
- Parent registration after hook registration without duplicate creation event.

Intentional changes:

- One bootstrap ID binds to one session.
- Copied bootstrap in another session is silent after binding.
- Distinct IDs still register distinct sessions despite identical other fields.
- thread.id is no longer a Codex session UUID.
- session_id and parent_session_id are the agtask-owned Codex identity fields.
- Old bootstrap and CLI vocabulary end at the hard-cut boundary.

## Risks and Mitigations

1. Title generator reaches hook first.
- Require primary-first as a whole-feature release gate; if contradicted on any
  supported surface, redesign the entire fallback around two-phase binding or
  a Codex primary-session signal.

2. Parent and hook race.
- BEGIN IMMEDIATE, PK/unique constraints, and one reconciliation state machine.

3. Logical ID sent to Codex.
- Distinct names and assertions at every action, route, and link.

4. Hard cut strands old queued children.
- Deploy emitter/parser together and reject old trailers rather than guess.

5. Two UUIDs confuse operators.
- Exact JSON/CLI names; --id only logical, --session-id only Codex identity.

## Acceptance Criteria

- thread.id is canonical UUIDv4 generated once before child creation;
  thread.session_id is nonempty and unique.
- parent_session_id replaces parent_thread_id in agtask-owned current contracts.
- rollout.thread_id references logical thread.id.
- Canonical v2 bootstrap requires id and parent_session_id.
- Parent/hook registration converge for parent-first, hook-first, and fallback.
- Bound ID from another session writes nothing and emits no tracked/action context.
- A tracked valid exact pair with matching metadata commits only idempotent/new
  rollout work, returns ordinary tracked context and the title action, and
  returns the pin action exactly when **pin=true**.
- A tracked valid exact pair with immutable metadata/title conflict writes
  nothing, returns/emits nothing, and rolls back or performs a read-only no-op.
- A tracked session paired with an ID owned by another row writes/emits nothing;
  a tracked session paired with an unclaimed different ID is never rebound and
  likewise writes/emits nothing.
- Same-pair retry is idempotent; session reuse by another ID is rejected.
- Exact deterministic replay of the supplied production trace produces one
  correct real-child row and no losing-session context/actions.
- Shared-ID concurrency produces one row; distinct-ID concurrency produces all.
- Hooks/lifecycle resolve by session ID; rollouts persist by logical thread ID.
- Every explicit thread-targeting command uses the shared mutually exclusive
  **--id** / **--session-id** resolver; hooks always enter through session ID.
- Register/show/list/search/dashboard/detail expose both IDs; human output uses
  exactly **Task ID**, **Session ID**, and **Parent session ID**; Codex
  links/actions use session ID.
- Implementation hard-cuts from version 4 to version 5; older ledgers are
  refused without mutation.
- Merging status, **thread_project_merging_idx**, merge claims and their
  indexes/FKs, logical **project_merge_claim.owner_thread_id**, tokenized close,
  and **rollout.id INTEGER PRIMARY KEY** are preserved.
- data_model, architecture, flows, CLI, README, skill, affected scenarios, and
  focused tests are synchronized and passing.
- Verification states that the supplied real-first/title-generator-second
  production trace is the accepted external platform evidence, and does not
  claim the standalone harness proves desktop internal-session or queued-host
  ordering.
- If primary-before-title ordering is contradicted on any supported surface,
  the entire unclaimed-ID fallback does not ship and must be redesigned.

## Open Questions

None in the data model. Primary-before-copy ordering remains an external
platform contract evidenced by the supplied production trace, not a property
the repository's standalone integration harness can establish. Contrary host
evidence is a release blocker and triggers the redesign policy above.
