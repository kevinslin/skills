# Integrator Review Workflow

Use this workflow to review learnings generated by `ag-learn` and decide which recommendations to integrate into a skill, codebase, or project process.

## Goals

- Make agents more effective over time.
- Make code and workflows easier to maintain.
- Reduce entropy through fewer special cases, less policy sprawl, and less duplicated logic.

## Inputs

- Primary: `ag-learn` outputs, for example files under `~/.llm/skills/learn`.
- Secondary: related PR context, incident notes, or repeated review findings.

If the input is not in `ag-learn` format, normalize each recommendation into:

- problem
- recommendation
- evidence
- scope

## Decision Framework

For each recommendation, score `0-3` on each heuristic:

1. Actionability and testability
   - Can this be implemented as concrete edits or process steps with a clear validation method?
2. Leverage and reuse
   - Will this improve many future tasks or sessions instead of only one local case?
3. Entropy delta
   - Does it remove duplication, branching, or exception paths and simplify long-term behavior?
4. Evidence strength
   - Is it backed by repeated failure patterns, explicit incidents, or clear measurable pain?
5. Reversibility and blast radius
   - Can the change be safely rolled back or limited if it does not work?
6. Maintenance load
   - Does it reduce net ongoing maintenance, or does it create long-term tax?

## Decision Rules

- `integrate_now`: high leverage, clear actionability, positive entropy delta, acceptable risk.
- `queue`: promising but blocked by missing evidence, dependencies, or timing.
- `reject`: low leverage, weak evidence, or likely to increase entropy or maintenance burden.

Reject or defer items that are vague, one-off, or complexity-increasing.

## Integration Target Selection

For accepted items, choose exactly one primary target:

- `skill`: improve prompts, workflows, scripts, or bundled references in skills.
- `code`: implement in source code, tests, or tooling.
- `project`: change process, docs, standards, or operating cadence.

Pick the smallest viable integration that captures most of the value.

## Output Format

Each recommendation must be in its own markdown section using:
`## Recommendation N: <short title>`

Do not merge multiple recommendations into one section.

For each recommendation section include all fields below:

- `Decision`: integrate_now | queue | reject
- `Primary target`: skill | code | project
- `Overall score`: `<x>/5`
- `Heuristic scores`: six scores (`0-3`) with short rationale
- `Benefit`: concrete expected upside if integrated
- `Integration plan`: concrete next steps
- `Validation`: how to verify outcome
- `Risk`: main failure mode and mitigation

Additional output requirements:

- Do not propose screenshot-specific process changes unless the user explicitly asks for them.
- Be explicit when rejecting recommendations that increase entropy.

Finish with:

- `Prioritized backlog`: `P0`, `P1`, `P2`
- `Drop list`: items explicitly not integrating now and why

Keep the review concise, specific, and decision complete.

## Template

Use this exact template shape:

```md
## Recommendation 1: <short title>

- Decision: <integrate_now|queue|reject>
- Primary target: <skill|code|project>
- Overall score: <0.0-5.0>/5
- Heuristic scores (0-3): Actionability <n>, Leverage <n>, Entropy delta <n>, Evidence <n>, Reversibility <n>, Maintenance load <n>
- Benefit: <specific benefit>
- Integration plan: <concrete steps>
- Validation: <how to verify>
- Risk: <main risk + mitigation>

## Recommendation 2: <short title>

- Decision: <integrate_now|queue|reject>
- Primary target: <skill|code|project>
- Overall score: <0.0-5.0>/5
- Heuristic scores (0-3): Actionability <n>, Leverage <n>, Entropy delta <n>, Evidence <n>, Reversibility <n>, Maintenance load <n>
- Benefit: <specific benefit>

```
